## 2. 本期目标

在 macOS 上完成一个无需注册、无需项目方服务器的最小消费决策闭环：

```text
录入商品
  → 补充购买动机
  → 读取本地预算、规则与相关历史
  → 调用用户配置的模型
  → 展示结构化建议
  → 用户记录最终决定
  → 冷静期后反馈
  → 本地历史参与下一次分析
```

本期重点不是覆盖所有购物平台，而是验证三个核心假设：

1. 用户是否愿意在购买前主动提交一次分析；
2. 结构化建议是否能影响购买决定；
3. 本地历史是否能让后续建议更贴合个人情况。

macOS 是第一验证平台，不是独立技术分支。项目从第一天采用 Flutter，共享领域模型、数据层、Agent 编排、模型客户端和绝大部分界面逻辑，为短期发布 Android / iOS 做准备。

## 3. 成功标准

### 3.1 功能成功标准

- 用户可以在 10 分钟内完成首次模型配置和第一次分析；
- 不注册账号也能使用全部 MVP 功能；
- API Key 不以明文写入数据库或日志；
- 商品无法自动解析时，用户仍可手动完成录入和分析；
- 模型返回异常时不会产生错误决策记录；
- 应用重启后预算、规则、历史、提醒和反馈保持完整；
- 用户可以导出和彻底清除本地数据。

### 3.2 产品验证指标

测试期只保存在本地，不上传遥测数据。应用在“本地统计”页面展示：

- 累计分析次数；
- 购买、等待、放弃、寻找替代的次数；
- 冷静期后改变决定的次数；
- 已完成购买后反馈的比例；
- 不同商品分类的后悔记录；
- 被后续分析实际引用的历史记录数量。

## 4. 用户与使用环境

### 4.1 首期用户

- 项目开发者本人及少量愿意自行配置 API 的测试用户；
- 使用 macOS 进行日常工作；
- 能从手机或桌面购物页面复制商品链接、文本或截图；
- 接受数据仅保存在当前 Mac；
- 理解模型建议不等于财务建议。

### 4.2 首期录入方式

按优先级实现：

1. 应用内粘贴商品链接；
2. 粘贴包含商品信息的文本；
3. 拖入或选择商品截图；
4. 手动填写商品名称、价格和平台；
5. macOS 分享扩展作为 P1 增强项，不阻塞首版闭环。

### 6.1 默认首页

首次启动进入配置引导；配置完成后默认进入“新建分析”。

主界面顶部始终显示：

- 当前月份；
- 本月非必要消费预算；
- 已确认购买金额；
- 剩余预算；
- “新建分析”主按钮。

## 7. 核心流程

### 7.1 首次启动

1. 展示产品说明：数据本地保存、请求直接发送到用户模型服务；
2. 用户确认隐私说明；
3. 进入模型配置；
4. 输入 Base URL、API Key、模型名称；
5. 点击“测试连接”；
6. 测试成功后设置币种和月度非必要消费预算；
7. 可选添加一条消费规则；
8. 进入新建分析页。

退出引导后允许从设置重新配置。模型连接未成功时，也允许先进入应用并手动保存商品草稿，但不能发起分析。

### 7.2 新建分析

#### 步骤 A：导入商品

页面提供一个统一输入区：

- 粘贴链接或商品描述；
- 拖入截图；
- 选择图片；
- 点击“手动输入”。

输入后应用尝试得到：

- 商品名称；
- 当前价格；
- 原价（可选）；
- 平台（可选）；
- 商品链接（可选）；
- 商品图片（可选）；
- 商品分类（可选）。

解析失败不能中断流程，直接进入手动确认页。

#### 步骤 B：确认商品

必填：

- 商品名称；
- 当前价格；
- 币种，默认继承用户设置。

选填：

- 平台；
- 分类；
- 链接；
- 图片；
- 备注。

价格必须大于或等于 0。价格为 0 时提示“免费商品不会参与预算计算”，仍允许继续。

#### 步骤 C：补充意图

三个问题中前两个必填：

1. 为什么想买？
2. 预计多久使用一次？
3. 如果不买，会有什么影响？（选填）

同时展示：

- 本月剩余预算；
- 商品占剩余预算比例；
- 命中的消费规则；
- 检索到的相似历史数量。

#### 步骤 D：生成建议

用户点击“开始分析”后：

1. 客户端构建最小必要上下文；
2. 显示即将发送的数据摘要；
3. 调用模型；
4. 校验 JSON 结构；
5. 成功后生成决策卡；
6. 失败时显示可理解的错误并允许重试。

不得在重试时重复创建商品和决策记录。

### 7.3 用户决定

决策卡底部提供四个等权操作：

- 仍然购买；
- 进入冷静期；
- 放弃购买；
- 寻找替代。

选择“仍然购买”时：

- 询问是否将金额计入本月非必要消费；
- 默认计入；
- 保存决定时间。

选择“进入冷静期”时：

- 默认采用模型建议时长；
- 用户可以修改为 1 小时、24 小时、48 小时、7 天或自定义；
- 创建 macOS 本地通知；
- 商品进入“冷静清单”。

选择“放弃购买”时直接保存，可选填写主要原因。

选择“寻找替代”时保存状态，允许记录替代方案，但 P0 不自动搜索商品。

### 7.4 冷静期结束

本地通知文案：

> 你之前想买「{商品名称}」。现在还想买吗？

用户打开应用后选择：

- 仍想购买；
- 继续等待；
- 已不想购买；
- 已经购买。

每次状态变化作为事件保存，不覆盖此前决定。

### 7.5 购买后反馈

用户确认购买后，默认在 7 天后创建反馈提醒；用户可以修改或关闭。

反馈字段：

- 实际使用频率：未使用、偶尔、经常；
- 是否符合预期：1–5 分；
- 后悔程度：不后悔、有一点、明显后悔；
- 主要原因：可多选并补充文字；
- 是否加入“已拥有物品”。

## 8. 功能需求

### FR-01 模型配置

字段：

- 配置名称；
- Base URL；
- API Key；
- 模型名称；
- 是否支持图片；
- Temperature，默认 0.2；
- 超时时间，默认 60 秒。

要求：

- P0 只启用一个当前配置；
- API Key 保存到 macOS Keychain；
- 设置页只显示脱敏后的 Key；
- 用户编辑 Base URL 或模型后，配置状态变为“未测试”；
- 连接测试成功后记录能力和时间，不记录请求正文；
- 支持清除密钥。

验收：

- 正确配置可以完成测试；
- 401、超时、网络不可达和模型不存在必须显示不同错误；
- 日志、数据库和导出文件中搜索不到完整 API Key。

### FR-02 商品导入

要求：

- 识别输入类型：URL、普通文本、图片；
- URL 首期只做通用元信息提取，不承诺站点专用解析；
- 普通文本可通过模型提取候选商品信息；
- 截图采用三级降级链路，不要求用户模型必须支持图片；
- 自动结果必须进入确认页；
- 原始输入保存在本地，用户可删除。

截图处理顺序：

1. **多模态模型**：当前模型支持图片且用户授权时，将图片直接发送给模型提取候选商品信息；
2. **本地 OCR + 文本模型**：模型不支持图片或用户不允许发送图片时，在设备本地提取文字，再将文字交给用户的文本模型整理；
3. **手动补全**：OCR 不可用、结果不足或模型调用失败时，保留截图并让用户填写商品名称与价格。

本地 OCR 的原始结果必须在发送给文本模型前展示或允许用户确认。无论采用哪条链路，最终商品字段都必须由用户确认。

验收：

- URL、文本、图片和手动录入均可进入同一个确认流程；
- 自动提取失败时不丢失原输入；
- 用户可以修改全部自动提取字段。
- 仅配置纯文本模型时，截图仍可通过本地 OCR 或手动补全完成分析；
- 用户拒绝发送图片后，不得以其他方式将图片二进制上传到模型服务。

### FR-03 预算管理

字段：

- 月份；
- 非必要消费预算；
- 已确认购买金额；
- 剩余金额。

规则：

- 预算按自然月保存；
- 新月份默认继承上月预算额度，不继承已消费额；
- 只有用户确认计入预算的购买才增加已消费额；
- 用户可修正金额，并保留修改时间。

验收：

- 跨月后生成新的预算记录；
- 删除购买记录时，预算金额同步重新计算；
- 预算不足不阻止用户选择购买。

### FR-04 消费规则

规则字段：

- 名称；
- 规则描述；
- 适用分类，可为空；
- 最低金额，可为空；
- 建议等待时长，可为空；
- 是否启用。

示例：

- 数码产品超过 500 元至少等待 48 小时；
- 本月剩余预算不足 20% 时不购买非必需品；
- 同类物品已有两个时优先放弃。

验收：

- 用户可以新增、编辑、停用和删除规则；
- 分析前展示本次命中的规则；
- 模型上下文只包含启用且相关的规则。

### FR-05 历史检索

P0 使用本地可解释检索，不引入向量数据库。

检索条件：

- 相同或相近分类；
- 名称关键词；
- 相近价格区间；
- 是否购买；
- 是否后悔；
- 最近时间优先。

最多向模型发送 5 条相关决策和 5 条已确认规律。每条历史只发送与当前判断相关的摘要。

验收：

- 决策卡可以展开查看“本次引用的个人历史”；
- 没有历史时明确标注“本次为通用分析”；
- 删除历史后，后续分析不再引用该记录。

### FR-06 AI 分析

模型输入包含：

- 当前商品；
- 用户购买动机和使用预期；
- 当前预算摘要；
- 命中的消费规则；
- 相关历史摘要；
- 已确认个人规律；
- 输出 JSON Schema 和安全约束。

模型不得接收：

- API Key；
- 与本次判断无关的完整历史；
- 本地绝对文件路径；
- 已被用户删除的数据。

输出枚举：

- `buy`；
- `wait`；
- `skip`；
- `alternative`；
- `insufficient_data`。

风险等级：`low`、`medium`、`high`。

信心等级：`low`、`medium`、`high`。

验收：

- 输出通过 JSON Schema 校验后才能展示；
- 缺少必填字段时自动进行一次格式修复请求；
- 修复仍失败时显示错误，不伪造默认建议；
- 模型结论必须附带理由和信息不足项；
- 模型不得返回自动执行动作。

### FR-07 决策卡

展示顺序：

1. 建议结论；
2. 风险和信心；
3. 一句话摘要；
4. 最多三条主要理由；
5. 预算影响；
6. 替代方案；
7. 缺失信息；
8. 引用的个人历史；
9. 用户决定按钮。

事实、用户输入和模型推断使用不同视觉标记。

验收：

- 用户无需阅读长文即可理解建议；
- 所有结论均可追溯到输入或模型推断；
- `insufficient_data` 不显示为购买或放弃建议。

### FR-08 历史与状态

历史列表支持：

- 按时间倒序；
- 按状态筛选；
- 按分类筛选；
- 关键词搜索；
- 查看详情；
- 修改最终状态；
- 删除记录。

状态采用事件记录，至少包括：

```text
draft
analyzed
waiting
intend_to_buy
purchased
skipped
seeking_alternative
feedback_completed
```

验收：

- 状态变化保留时间线；
- 修改最终状态不覆盖原 AI 建议；
- 删除前二次确认，并说明关联数据范围。

### FR-09 提醒

提醒类型：

- 冷静期结束；
- 购买后反馈。

要求：

- 使用 macOS 本地通知；
- 首次创建提醒时请求通知权限；
- 用户拒绝权限后仍在应用内展示到期状态；
- 修改或删除决策时同步更新通知；
- 不使用服务器推送。

### FR-10 个人规律

P0 只生成有明确证据的简单规律，例如：

- 某分类累计至少 3 次购买，其中至少 2 次明显后悔；
- 某分类累计至少 3 次进入冷静期，其中多数最终放弃；
- 某价格区间的已购商品使用率持续偏低。

规律状态：

- 待确认；
- 已确认；
- 已忽略。

只有已确认规律默认进入模型上下文。待确认规律在“个人规律”页面展示证据，用户可以确认、编辑或忽略。

### FR-11 数据管理

P0 支持：

- 导出 JSON 数据包；
- 导出前展示包含的数据；
- 默认包含图片，可由用户取消；
- 永远不包含 API Key；
- 清除全部业务数据；
- 单独清除模型配置和 Key；
- 打开本地数据目录。

数据导入列为 P1，避免首版承担版本迁移和冲突处理。

## 9. 页面需求

### 9.1 新建分析页

空状态包含：

- 拖放区域；
- 粘贴按钮；
- 选择图片；
- 手动输入；
- 最近一次分析入口；
- 当前预算摘要。

处理中显示具体阶段：读取输入、提取信息、检索历史、等待模型、校验结果。禁止只显示无限旋转图标。

### 9.2 冷静清单

每个项目显示：

- 商品名称和价格；
- 剩余时间；
- 原建议；
- 当前状态；
- 立即重新判断；
- 修改等待时间。

### 9.3 决策历史

列表行显示商品、价格、日期、AI 建议、用户决定和反馈状态。详情页展示完整时间线和本次引用的上下文。

### 9.4 个人规律

分为“待确认”和“已确认”。每条规律必须展示支持它的记录数量和可打开的证据列表。

### 9.5 本地统计

仅展示基础数字和简单分组，不做复杂图表。明确标注“所有统计均在本机计算”。

### 9.6 设置

- 模型服务；
- 预算与币种；
- 消费规则；
- 通知偏好；
- 数据管理；
- 隐私说明；
- 开源许可证和仓库入口。

## 10. 本地数据模型

建议核心表：

| 表 | 作用 |
| --- | --- |
| `app_settings` | 币种、通知和通用偏好 |
| `model_configs` | 非敏感模型配置，Keychain 仅保存密钥引用 |
| `monthly_budgets` | 月度预算与调整记录 |
| `spending_rules` | 用户消费规则 |
| `products` | 商品基础信息 |
| `product_assets` | 截图与图片的本地相对路径 |
| `decision_inputs` | 购买动机和使用预期 |
| `analyses` | 模型原始响应、解析结果和 Schema 版本 |
| `decision_events` | 用户决定及状态时间线 |
| `feedback` | 购买后使用和后悔反馈 |
| `owned_items` | 用户已拥有物品 |
| `agent_memories` | 个人规律、证据和确认状态 |
| `reminders` | 本地提醒计划 |

所有业务表使用本地 UUID。时间统一保存为 UTC，界面按系统时区显示。金额使用整数最小货币单位，避免浮点误差。

### 10.1 跨端架构约束

Flutter 项目按能力分层，平台差异封装在适配层：

```text
UI / 页面状态
      ↓
领域用例与 Agent 编排
      ↓
Repository 接口
      ↓
SQLite / 安全存储 / 文件 / 通知 / 分享适配器
```

要求：

- 商品、预算、决策、反馈和 Agent 记忆使用平台无关的数据模型；
- SQLite Schema 和迁移在 macOS、Android、iOS 保持一致；
- Keychain 与 Android Keystore 通过统一安全存储接口访问；
- 本地通知、文件选择、图片导入和系统分享使用平台适配器；
- OCR 使用统一接口：macOS/iOS 优先采用 Apple Vision，Android 采用可本地运行的 OCR 适配器；
- 模型客户端、JSON Schema 校验和上下文组装不得依赖 macOS API；
- 业务层测试必须能够在无桌面窗口和无平台插件的环境中运行；
- macOS 特有能力不能直接进入领域层。

### 10.2 平台推进顺序

1. macOS：验证配置、商品录入、模型分析、本地历史和反馈闭环；
2. Android：作为首个移动端发布平台，优先验证系统分享接收、后台限制、本地 OCR、本地通知和安全存储；
3. iOS：在 Android 闭环稳定后，验证 Share Extension、App Group 数据交接、Vision OCR、Keychain 和通知权限；
4. 三端统一数据 Schema、提示词版本和决策结果结构。

移动端的“从购物 App 分享给购物守护者”是正式产品核心路径。macOS 的粘贴、拖放和手动录入仅是低成本验证入口。

## 11. 非功能需求

### 11.1 隐私

- 默认无遥测、无崩溃日志上传；
- 不请求通讯录、相册全量权限或浏览器历史；
- 图片通过系统文件选择器按次授权；
- 调用模型前展示将发送的数据类型；
- 用户可关闭图片发送，只提交确认后的文字字段。

### 11.2 安全

- Keychain 保存 API Key；
- 数据库和日志不记录密钥；
- Base URL 仅允许 `https`，本地开发地址和局域网地址需用户显式开启；
- 商品文本、网页元信息和图片 OCR 均视为不可信输入；
- 提示词明确要求忽略商品内容中的指令；
- 外部链接在系统浏览器打开前显示目标域名；
- 不自动下载或执行商品链接中的文件。

### 11.3 性能

- 冷启动目标小于 3 秒；
- 本地页面切换目标小于 300 毫秒；
- 非模型操作不得因网络不可用而阻塞；
- 模型请求超过用户配置时间后可取消；
- 图片在保存前压缩，原图保留行为由用户设置决定。

### 11.4 可恢复性

- 分析过程中异常退出后保留草稿；
- 数据库写入采用事务；
- Schema 版本随导出文件保存；
- 数据库迁移失败时不覆盖原文件，并给出备份路径。

### 11.5 可访问性

- 支持 macOS 深色模式；
- 支持系统字体缩放；
- 核心操作可通过键盘完成；
- 状态不只依赖颜色表达；
- 通知和错误信息提供可读文本。

## 12. Agent 上下文规范

### 12.1 上下文组装顺序

1. 系统角色与安全边界；
2. 输出 Schema；
3. 当前商品和用户输入；
4. 预算摘要；
5. 命中规则；
6. 最多 5 条相关历史；
7. 最多 5 条已确认规律；
8. 不确定信息列表。

### 12.2 事实标签

发送给模型的每项信息标记来源：

- `user_input`：用户主动输入；
- `local_record`：本地历史；
- `parsed`：从链接、文本或图片提取但已由用户确认；
- `derived`：客户端计算，例如预算比例；
- `unknown`：缺失信息。

### 12.3 决策约束

- 不编造价格历史、商品评价或用户资产；
- 不以限时促销文案作为购买理由；
- 不因用户预算充足就默认建议购买；
- 不使用羞辱、恐吓或医疗化表述；
- 证据冲突时降低信心等级；
- 信息不足时返回 `insufficient_data`；
- 建议只影响展示，不产生外部动作。

## 13. 错误与边界状态

必须覆盖：

- 未配置模型；
- API Key 无效；
- Base URL 无法访问；
- 模型不存在；
- 请求超时或被用户取消；
- 模型不支持图片；
- 模型返回非 JSON；
- JSON 缺少字段或枚举非法；
- 链接无法访问或解析；
- 图片无法读取；
- 数据库空间不足或写入失败；
- 通知权限被拒绝；
- 预算未设置；
- 没有历史数据；
- 商品价格超过预算；
- 删除了正在等待的商品。

每个错误必须说明：发生了什么、数据是否已保存、用户下一步可以做什么。

## 14. 开发与测试计划

### Milestone 0：技术样机

目标：证明关键技术可行，不做完整 UI。

- macOS 应用骨架；
- SQLite 读写；
- Keychain 读写；
- OpenAI-compatible 连接测试；
- JSON Schema 校验；
- 本地通知；
- 图片拖放与沙盒文件访问。
- 截图的多模态、本地 OCR 和手动补全三级降级链路；
- Flutter macOS 工程和跨端分层骨架；
- Android / iOS 空壳工程能够编译，关键依赖不存在明确平台阻塞。

退出条件：所有能力均有最小自动测试或可重复手工测试步骤。

### Milestone 1：单次决策

- 首次配置；
- 商品录入和确认；
- 预算与规则；
- 模型分析；
- 决策卡；
- 本地保存。

退出条件：全新安装后可以离线完成配置之外的所有本地操作，并通过自带模型完成一次分析。

### Milestone 2：长期闭环

- 冷静清单；
- 本地通知；
- 状态时间线；
- 购买后反馈；
- 历史检索；
- 基础个人规律。

退出条件：同一用户连续完成 5 次决策后，新分析可以引用相关历史。

### Milestone 3：开源准备

- 数据导出与清除；
- 隐私和安全检查；

- 示例配置和测试数据；
- 开源协议、贡献指南和 Issue 模板。

## 15. 发布前检查

- 在一台未配置开发环境的 Mac 上完成安装；
- 无 API Key 时应用可正常打开和管理本地数据；
- 网络断开时不影响历史、规则和草稿；
- 导出包不含 Keychain 密钥或绝对文件路径；
- 清除数据后数据库、图片、通知和密钥全部清理；
- 日志中不存在 API Key、完整请求头和敏感输入；
- 所有模型结论均显示“AI 辅助建议”；
- README 明确项目方不提供在线服务；
- README 明确不同模型服务可能收集用户发送的数据；
- 至少完成一次从首次配置到购买后反馈的端到端测试。

## 16. 本期需要确认的产品决策

以下决策不会阻塞技术样机，但应在 Milestone 1 前确定：

已确定：Flutter；macOS 首测；Android 优先发布、随后 iOS；图片能力采用多模态模型、本地 OCR、手动补全三级降级；Apache License 2.0。

仍需确定：

1. 最低支持的 macOS、Android 和 iOS 版本；
2. 是否允许连接 `http://localhost` 和局域网模型；
3. 默认冷静期和购买后反馈时间；
4. 商品分类使用固定枚举还是允许用户自定义；
5. 原始模型响应保存多久，是否允许用户关闭保存；
6. 首版数据导出是否包含商品图片；
7. 应用名称和图标是否沿用“购物守护者”；
8. 首批测试用户的模型服务与典型商品来源。

## 17. 结论

macOS v0.1 的唯一任务，是以尽可能少的工程复杂度验证“个人消费 Agent”是否真的能改善用户决策。

首版产品必须做到：本地可信、输入简单、建议可解释、结果能反馈。商品平台覆盖、自动抓取、复杂比价和真正的模型训练，都不应早于这个闭环得到验证。
