# 购物守护者 MRD v0.2

> 本地优先、用户自带模型的个人消费决策 Agent

## 文档信息

| 项目 | 内容 |

| 版本 | v0.2 |

## 一、产品概述

### 1.1 一句话定位

购物守护者是一款开源、本地优先的个人消费决策工具。用户将想购买的商品链接、截图或文字分享给应用，应用结合用户的预算、个人规则和历史消费反馈，调用用户自己的模型服务，生成结构化购买建议。

### 1.2 产品愿景

电商平台持续缩短用户从“看到商品”到“完成付款”的时间，购物守护者则在两者之间增加一次属于用户自己的理性判断。

产品不负责导购，不依赖销售佣金，也不以促成交易为目标。它只帮助用户回答一个问题：

> 这件东西适不适合现在的我？

### 1.3 核心原则

1. **本地优先**：预算、商品、决策、反馈和 Agent 记忆默认保存在用户设备上。
2. **用户自带模型**：用户配置自己的 API Key、模型和服务地址，分析请求不经过项目方服务器。
3. **无需注册**：核心功能不依赖账号系统或在线后台。
4. **开源透明**：数据处理、提示词和决策逻辑可查看、可修改、可审计。
5. **辅助而非控制**：产品提供判断依据，但不替用户付款，也不强制阻止购买。
6. **允许不确定**：信息不足时明确提示，不伪造价格、历史数据或精确概率。

## 二、问题与机会

### 2.1 用户问题

现有购物工具主要解决“在哪里买更便宜”，但很少解决“是否应该购买”。用户通常面临以下问题：

- 在直播、短视频、限时优惠等刺激下快速下单；
- 无法在购买当下准确评估实际使用价值；
- 忘记自己已经拥有功能相近的物品；
- 只关注商品价格，忽略本月预算和储蓄目标；
- 买后后悔，却没有积累成下一次可用的经验；
- 不愿把完整消费数据上传给另一个在线平台。

### 2.2 市场空白

比价、优惠和导购产品的利益通常与成交一致，而用户真正缺少的是一个利益完全站在自己一边的决策工具。

购物守护者的机会不在于拥有更多商品，而在于长期积累用户自己的消费决策数据，并逐渐识别：

- 哪些品类最容易冲动购买；
- 哪个价格区间最容易后悔；
- 哪些购买理由最终经常被证明不成立；
- 等待多长时间后，购买意愿通常会消失；
- 哪些商品虽然价格高，但确实被长期使用。

## 三、目标用户与场景

### 3.1 核心用户

- 有网购习惯，主要在手机 App 中完成商品浏览和购买；
- 知道自己存在冲动消费，但不想使用复杂记账软件；
- 愿意在重要购买前多花几十秒进行判断；
- 重视隐私，偏好本地存储和自主管理数据；
- 能够配置模型 API，或愿意按照引导完成配置。

早期开源用户可以偏向开发者与 AI 工具用户，但产品交互不应假设所有用户都理解模型参数。

### 3.2 典型场景

1. 用户在淘宝、京东、闲鱼、小红书等 App 看到商品，将商品分享给购物守护者。
2. 平台无法直接分享时，用户复制链接并粘贴到应用。
3. 链接无法解析时，用户导入截图并确认商品名称与价格。
4. 用户一次导入多个商品，让 Agent 帮助排序和取舍。
5. 冷静期结束后，应用提醒用户重新判断是否仍想购买。
6. 购买一段时间后，用户记录使用频率和后悔程度，形成个人经验。

## 四、产品边界

### 4.1 产品负责

- 接收商品链接、截图和文字；
- 尽可能提取商品的基础信息；
- 维护本地预算、消费规则、物品和决策历史；
- 检索与当前商品相关的个人历史；
- 调用用户配置的模型服务；
- 展示结构化建议并记录用户决定；
- 在冷静期和购买后进行提醒；
- 支持本地数据导入、导出和彻底删除。

### 4.2 产品不负责

- 不提供官方托管的模型服务；
- 不建设账号、云数据库和运营后台；
- 不绑定银行卡或读取支付账户；
- 不自动下单，也不拦截其他 App 的付款操作；
- 不承诺全平台实时比价或完整历史价格；
- 不依赖未授权爬虫绕过平台限制；
- 不向用户推荐带佣金的商品；
- MVP 不进行模型微调或训练基础模型。

## 五、核心用户流程

### 5.1 首次配置

1. 用户阅读本地存储和模型请求说明；
2. 配置模型服务：Base URL、API Key、模型名称；
3. 测试模型连接；
4. 设置每月非必要消费预算；
5. 可选填写消费原则，例如“数码产品至少等待 48 小时”；
6. 完成本地数据库初始化。

### 5.2 单商品决策

1. 用户分享链接、粘贴链接、导入截图或手动输入商品；
2. 应用解析商品名称、价格、平台、图片和链接；
3. 用户确认或修正信息；
4. 用户回答少量关键问题：
   - 为什么想买？
   - 预计多久使用一次？
   - 不买会产生什么影响？
5. 应用从本地检索相关预算、物品和历史决策；
6. 调用用户模型生成结构化建议；
7. 用户选择“购买”“等待”“放弃”或“寻找替代”；
8. 应用保存本次决策，并按需要创建提醒。

### 5.3 决策反馈

1. 等待期结束后询问用户是否仍想购买；
2. 对已购买商品，在设定时间后询问：
   - 实际使用频率；
   - 是否达到预期；
   - 是否后悔；
   - 后悔或满意的主要原因；
3. 将反馈写入本地历史；
4. 更新可复用的个人消费规律。

## 六、决策框架

Agent 从以下六个维度进行分析。

### 6.1 需求真实性

- 商品解决什么具体问题？
- 不购买是否会造成实际影响？
- 需求是长期存在，还是刚被内容或促销触发？

### 6.2 使用价值

- 预计使用频率和使用周期；
- 预估单次使用成本；
- 是否存在买来闲置的高风险信号。

### 6.3 重复与替代

- 用户是否已拥有同类物品；
- 现有物品为什么不能满足需求；
- 是否可以借用、租赁、维修、购买二手或选择低价替代。

### 6.4 价格与时机

- 当前价格与用户提供的参考价格；
- 是否值得等待促销或价格下降；
- 缺少可靠价格数据时，不生成虚构的历史最低价。

### 6.5 财务影响

- 商品价格占本月非必要消费预算的比例；
- 购买后的预算余额；
- 是否影响用户设置的储蓄或消费目标；
- 最近是否已经发生多次非必要购买。

### 6.6 后悔风险

根据行为信号和个人历史判断低、中、高风险，不输出伪精确百分比。可用信号包括：

- 第一次看到商品后很快提交分析；
- 来自直播、短视频或限时促销；
- 使用“最后一天”“马上涨价”等紧迫理由；
- 过去同类商品使用率低或后悔率高；
- 家中已有同类物品；
- 价格明显超出剩余预算。

## 七、决策结果

### 7.1 结果类型

- `buy`：当前条件下可以购买；
- `wait`：建议进入冷静期后重新评估；
- `skip`：目前没有足够购买价值；
- `alternative`：优先寻找替代方案；
- `insufficient_data`：信息不足，无法可靠判断。

### 7.2 输出结构

模型必须返回结构化 JSON，由客户端负责校验与展示。

```json
{
  "recommendation": "wait",
  "risk_level": "high",
  "confidence": "medium",
  "summary": "建议等待 48 小时后重新判断",
  "reasons": [
    "商品价格将占用本月剩余非必要消费预算的 82%",
    "你已经拥有两个功能相近的物品",
    "过去三次同类购买中有两次被标记为后悔"
  ],
  "alternatives": [
    "继续使用已有物品",
    "寻找二手商品",
    "价格低于 299 元时重新评估"
  ],
  "cooldown_hours": 48,
  "missing_information": []
}
```

### 7.3 展示原则

- 先给结论，再给最重要的三条理由；
- 明确区分事实、用户数据和模型推断；
- 不对消费行为进行羞辱或道德评判；
- 始终允许用户作出与建议不同的决定；
- 模型不可用时，保留商品和用户输入，允许稍后重试。

## 八、个人 Agent 与本地记忆

### 8.1 MVP 中“训练”的含义

MVP 不训练或微调基础模型。“个人 Agent”通过本地记忆和历史检索实现个性化：

```text
当前商品与购买理由
        +
预算和个人消费规则
        +
相关个人物品
        +
相似品类的历史决策与反馈
        +
已确认的个人消费规律
        ↓
用户配置的模型
        ↓
结构化购买建议
```

### 8.2 记忆类型

1. **用户主动设置的规则**：例如“超过 500 元至少等待两天”。
2. **客观历史记录**：购买、等待、放弃、使用和后悔反馈。
3. **系统推断的规律**：例如“数码配件历史使用率偏低”。
4. **用户确认的规律**：由系统提出、用户确认后长期使用。

系统推断不能自动变成永久规则，必须允许用户查看、修改、确认和删除。

### 8.3 检索策略

MVP 优先采用简单、可解释的本地检索：

- 商品分类匹配；
- 名称和关键词匹配；
- 价格区间匹配；
- 决策时间和结果筛选；
- 是否后悔、使用频率等标签筛选。

当数据量确实增长后，再考虑本地向量检索，不作为第一版前置依赖。

## 九、本地数据设计

MVP 使用 SQLite 保存结构化数据，商品图片和导入截图保存为本地文件，数据库只保存文件路径。

### 9.1 核心实体

| 实体 | 主要字段 |
| --- | --- |
| ModelConfig | provider、baseUrl、model、能力配置；API Key 单独安全存储 |
| UserProfile | 月度预算、币种、提醒偏好 |
| SpendingRule | 规则内容、适用品类、金额阈值、等待时长 |
| Product | 名称、价格、平台、链接、图片、分类、来源类型 |
| Decision | 购买理由、模型建议、用户决定、风险等级、创建时间 |
| Feedback | 是否购买、使用频率、满意度、后悔程度、原因 |
| OwnedItem | 已拥有物品、购入时间、状态、使用频率 |
| AgentMemory | 规律内容、证据来源、状态、用户是否确认 |
| Reminder | 提醒类型、触发时间、完成状态 |

### 9.2 数据生命周期

- 用户可以单独删除某次决策及关联图片；
- 用户可以清空全部数据；
- 用户可以导出不含 API Key 的数据包；
- 导入前展示数据范围并检查版本兼容性；
- 删除本地数据后不保留项目方云端副本，因为项目方不接收这些数据。

## 十、模型接入与安全

### 10.1 接入方式

MVP 优先支持 OpenAI-compatible API：

- Base URL；
- API Key；
- Model；
- 是否支持图片输入；
- 可选高级参数；
- 测试连接。

后续可根据社区需求增加 Ollama 等本地推理服务的专用适配。

### 10.2 安全要求

- API Key 使用系统 Keychain/Keystore 等安全存储，不写入 SQLite；
- 日志不得打印完整 API Key、请求头或敏感消费信息；
- 数据导出默认排除 API Key；
- 请求直接从客户端发往用户配置的模型服务；
- 每次发送前明确模型将接收的内容类型；
- 商品链接视为不可信输入，不执行其中携带的指令；
- 模型输出必须经过 JSON Schema 校验；
- 模型输出只是建议，不得触发支付、删除或其他高风险操作。

## 十一、商品导入策略

### 11.1 支持入口

按优先级支持：

1. 系统分享文本或链接；
2. 应用内粘贴链接；
3. 导入商品截图；
4. 手动输入商品名称和价格；
5. 后续支持购物车或清单的批量导入。

### 11.2 解析原则

- 解析器只负责尽力提取，不将成功解析作为完成决策的前提；
- 所有自动提取的信息必须允许用户修改；
- 跳转链接和短链接应限制重定向次数；
- 不使用需要绕过登录、验证码或平台限制的方案；
- 无法取得完整详情时，以用户确认后的最小信息继续分析；
- MVP 不承诺覆盖所有电商平台。

## 十二、MVP 功能范围

### 12.1 必须完成

1. 本地初始化，无需注册登录；
2. 模型服务配置与连接测试；
3. API Key 安全存储；
4. 分享、粘贴、截图和手动输入中的至少三种商品导入方式；
5. 商品信息确认与编辑；
6. 月度非必要消费预算和个人规则；
7. 相关历史记录检索；
8. 结构化 AI 决策卡；
9. 购买、等待、放弃、寻找替代四种用户决定；
10. 冷静期提醒；
11. 购买后反馈；
12. 本地决策历史；
13. 数据导出、导入和彻底删除；
14. 基础错误处理与隐私说明。

### 12.2 暂不实现

- 浏览器插件；
- 官方在线模型服务；
- 账号系统和多设备实时同步；
- 银行、支付宝或微信账单接入；
- 自动读取完整购物车；
- 全平台商品爬取和实时比价；
- 自动下单或强制阻止下单；
- 社区、排行榜和商品推荐；
- 广告、返佣和导购链接；
- 基础模型微调；
- 复杂向量数据库和多 Agent 编排。

## 十三、MVP 验收标准

### 13.1 功能验收

- 用户在不注册账号的情况下可以完成全部核心流程；
- 用户可以成功配置至少一种 OpenAI-compatible 模型服务；
- 导入失败时可以通过手动补全继续分析；
- 模型返回格式错误时应用能够提示并安全重试；
- 每次建议都能追溯使用了哪些用户数据和历史记录；
- 用户决定和后续反馈能够正确关联；
- 应用重启后本地数据保持完整；
- 导出文件不包含 API Key；
- 清除数据后，本地数据库、图片和安全存储中的密钥均被移除。

### 13.2 产品验证指标

开源 MVP 不追求注册量或交易额，优先验证：

- 用户是否愿意在购买前主动导入商品；
- 一次分析是否能在较少输入下完成；
- 用户是否理解并信任建议的理由；
- 进入冷静期后，有多少购买意愿消失；
- 用户是否愿意回来填写购买后反馈；
- 历史数据是否让后续建议明显更个性化；
- 用户能否独立完成模型配置和数据管理。

### 13.3 建议的早期成功信号

- 新用户在十分钟内完成模型配置和第一次分析；
- 常规商品分析需要用户填写的必答问题不超过三个；
- 冷静期结束后，用户能明确感知建议是否帮助了决策；
- 积累五次以上反馈后，系统可以展示至少一条有证据的个人消费规律。

## 十四、开源策略

### 14.1 仓库内容

- 应用源代码；
- 数据库迁移和数据结构说明；
- 默认提示词与 JSON Schema；
- 模型接入文档；
- 隐私和安全说明；
- 本地开发、构建和发布步骤；
- 贡献指南、Issue 模板和路线图；
- 示例数据，但不得包含真实用户消费信息或密钥。

### 14.2 开源原则

- 采用 Apache License 2.0，允许自由使用与分发，同时提供明确的专利授权条款；
- 核心功能不依赖项目方私有服务；
- 不在默认版本中加入商品返佣；
- 第三方商品解析器以模块形式维护，失效时不影响核心流程；
- 社区贡献的新模型适配必须遵守统一安全接口。

## 十五、技术方向建议

在正式技术选型前进行小型验证。当前建议：

- 客户端：Flutter，macOS 作为首个验证平台，共享业务层短期扩展 Android / iOS；
- 本地数据：SQLite；
- 密钥：系统 Keychain/Keystore；
- 图片：应用沙盒文件目录；
- 模型协议：OpenAI-compatible API；
- 模型输出：JSON Schema；
- 提醒：系统本地通知；
- 个性化：规则检索与历史检索，后续再评估本地向量搜索。

该技术方向不是产品成立的前提。进入开发前，应优先验证以下风险：

1. iOS 与 Android 的系统分享入口实现成本；
2. 常见分享文本和短链接的可解析程度；
3. 不同兼容模型返回结构化 JSON 的稳定性；
4. Flutter 在目标平台上的安全存储与本地通知体验；
5. 验证截图三级降级链路：多模态模型、本地 OCR + 文本模型、手动补全。

## 十六、开发阶段

### Phase 0：技术验证

- 验证系统分享入口；
- 验证一种链接解析和截图导入；
- 验证模型配置、直连和结构化输出；
- 验证 SQLite 与安全存储。

### Phase 1：最小决策闭环

- 商品录入与确认；
- 预算和消费规则；
- 单次模型分析；
- 决策卡与本地保存。

### Phase 2：反馈闭环

- 冷静期提醒；
- 购买后反馈；
- 历史记录与基础统计；
- 相关历史检索。

### Phase 3：个人 Agent

- 自动提出消费规律；
- 用户确认、编辑和删除记忆；
- 在新决策中引用相关历史证据；
- 优化提示词与上下文长度。

### Phase 4：开源发布

- 完善测试和异常处理；
- 数据导入导出；
- 隐私、安全和构建文档；
- 发布可复现构建与贡献指南。

## 十七、待验证问题

已确定：移动端先发布 Android，再适配 iOS；截图能力不得要求用户模型必须支持多模态；开源协议采用 Apache License 2.0。

仍待验证：

1. 商品图片授权采用“每次询问”还是允许用户记住对当前模型的选择？
2. 冷静期提醒的默认时间如何设计？
3. 用户需要录入多少财务信息，才能在价值与隐私之间取得平衡？
4. 个人物品库由用户主动维护，还是从已确认购买记录逐渐生成？
5. 是否需要兼容 Ollama 等局域网或本地模型服务？
6. 数据导出采用 JSON、ZIP，还是同时提供可读的 Markdown 报告？
7. 第一批测试用户是否愿意持续填写购买后反馈？

## 十八、版本结论

v0.2 不再将购物守护者定义为浏览器插件或在线消费服务，而是一个无需账号、无需项目方服务器、数据由用户掌控的开源个人工具。

MVP 的核心不是覆盖所有购物平台，也不是构建复杂的 AI 系统，而是完成一个可信的小闭环：

> 导入想买的商品 → 结合个人历史做出判断 → 记录真实结果 → 让下一次判断更了解用户。
